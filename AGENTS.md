# order-song Agent Guide

## Project Shape

`order-song` is a FastAPI + SQLite backend with a browser frontend for multi-user song ordering, room playback sync, playlists, NetEase import, Bilibili search/audio, and an admin panel.

- Backend: `backend/app/`
  - `main.py` creates the FastAPI app, registers routers, WebSocket, startup tasks, and static frontend serving.
  - `routers/` contains API endpoints grouped by feature: auth, rooms, queue/playback, playlists/trending, search, admin.
  - `models.py` contains SQLModel tables and enums. Treat these as database contract.
  - `schemas.py` contains Pydantic request/response models shared by routers.
  - `core/config.py` loads `config.toml` and `ORDER_SONG_*` env overrides.
  - `ws.py` owns room WebSocket membership and broadcast state.
- Frontend: `frontend/`
  - React + TypeScript source lives in `frontend/src/`.
  - `frontend/src/App.tsx` wires the app shell, auth, rooms, search, queue, playlists, admin, WebSocket, and data queries.
    It also owns shared card dialogs, internal scroll containers, playlist membership selection cards, the NetEase playlist import entry, and admin permission controls.
  - `frontend/src/hooks/useAudioController.ts` owns audio playback, aggressive timestamp-based sync, seek/next controls, and local volume normalization.
  - `frontend/src/lib/` contains API, storage, time, and track helpers.
  - `frontend/src/styles.css` contains the iOS 26-inspired liquid glass visual system.
  - `frontend/dist/` is generated by `npm run build` and is served by FastAPI in production.
  - `frontend/legacy-index.html` and `frontend/assets/` are the pre-React static frontend fallback. Keep them only as compatibility fallback unless intentionally removing the legacy app.

## Playback Sync Contract

Playback state responses and WebSocket `playback_updated` messages include:

- `playback_state`: persisted room playback row.
- `server_time` and `server_ts_ms`: server broadcast/snapshot timestamp.
- `effective_position_ms`: server-calculated current position at `server_ts_ms`.
- `current_track` and `ordered_by`: metadata for the current queue item.

`server_ts_ms` must be emitted as a UTC epoch millisecond value. Do not call `.timestamp()` on a naive UTC `datetime` without first marking it as UTC, because local timezone interpretation will make clients seek to the wrong position.

Frontend clients should compute live progress from `effective_position_ms + (Date.now() - server_ts_ms)` while playing, and should reconcile the local audio element aggressively when drift exceeds a small tolerance. Next-track requests should include `expected_queue_item_id`; the backend serializes room-level next operations and ignores stale duplicate next requests.

Bilibili tracks may arrive without `cover_url`; queue/playback and playlist item creation should best-effort fetch Bilibili video metadata and persist the video `pic` as the cover before broadcasting or displaying current-track data. The React frontend should prefer direct Bilibili image URLs with `referrerPolicy="no-referrer"` to keep cover traffic off the app server.

Playback audio URLs that point at upstream services should be exposed to the browser as `/api/tracks/{track_id}/stream` whenever possible. This keeps playback same-origin and avoids WebAudio silence caused by CORS restrictions, especially for NetEase Cloud Music outer URLs. The frontend audio controller should defensively rewrite playable NetEase/Bilibili/current-track absolute audio URLs to that same-origin stream when a track id is available.

## Frontend Interaction Notes

The room app uses fixed-height internal scroll areas so mobile song lists do not push the player bar off-screen. Search opens from a top-bar button next to the play-enabled switch; there is no persistent search dock above the library. The right queue column is available on wide screens; the main-column queue tab is only shown once that side column collapses. Search history appears when the search input is focused and hides after a search is executed. Destructive confirmations, playlist membership edits, playlist renaming, new playlist creation, and NetEase playlist import should use in-app card dialogs, not browser `alert`, `confirm`, or `prompt`.

Audio sync should tolerate small device/loading drift. The frontend currently allows about 3 seconds of drift and force-seeks on track/source changes, instead of re-seeking on every periodic room-state refresh.

## Database And Migration

Do not change existing tables, columns, enum values, or constraints casually. `backend/order_song.sqlite3` is an ignored local database, but user data may depend on the current schema.

If a migration is required:

- Prefer the existing lightweight, idempotent style in `backend/app/db.py`.
- Add only compatible `ALTER TABLE ... ADD COLUMN ...` migrations unless the user explicitly approves a destructive migration.
- Document the reason, default value, compatibility behavior, and verification steps in this file.

## Agent Maintenance Rule

Future agents must update this `AGENTS.md` in the same change whenever they modify module structure, API routes or payloads, schemas, database models or migrations, configuration keys, frontend source layout, build/start commands, playback synchronization behavior, or the visual design system.
